<Dropcap text="The repo for this site contains no package.json. No node_modules. No webpack, esbuild, vite, parcel, rollup, gulp, grunt, or fabric of any other generation. There is one build script. It is six hundred lines of Node, in a single file, and it uses two modules: <code>fs</code> and <code>path</code>." />

The first time I built it, I built it on a train, on an iPad, in a single-file editor with `node build.js` typed at a shell prompt. The second time, I built it on my phone — iSH, an emulator that runs Alpine Linux in a tab. It took 90 milliseconds. Both builds produced bytewise-identical output to the build that runs on GitHub Pages.

I want to talk about how that's possible, because I think the answer is interesting in a small way. Most of the writing in our industry assumes a tooling pyramid. You install a framework; the framework needs a bundler; the bundler needs a transformer; the transformer needs a compiler. By the time you ship a paragraph of writing, you have a quarter-gigabyte of dependencies and a six-line GitHub Actions file that knows how to bring them down before it can bring them up.

I do not, in general, mind that. There are jobs the pyramid is good at. But for a personal site — for somewhere I wanted to put essays, notes, projects, and the occasional album — it felt like a shape that didn't fit. So I tried the smallest version I could think of, and the smallest version turned out to be enough.

## The shape of it

Three directories matter. `src/` is where the source lives. `dist/` is where the build output goes. `.github/workflows/pages.yml` runs `node build.js` on every push to `main` and uploads `./dist` to GitHub Pages.

Inside `src/` there are five subdirectories you'll keep coming back to.

- **`src/content/`** holds the writing. It's mostly MDX (markdown with custom components in JSX-like syntax) plus three JSON files (`site.json`, `now.json`, `about.json`) for content that's structured rather than narrative. Each piece of writing — essay, study, project case, note, book, album, movie, podcast — is one MDX file with YAML frontmatter and a body.
- **`src/layouts/`** holds the HTML templates. There's one per page type: home, writing-index (essays + studies share it), article reader, project index + reader, notes index + reader, the per-kind cultural readers (reading, music, movies, podcasts, bookshelf), the unified `enjoying` paginated index, plus `now`, `about`, `learning`, `gear`, `colophon`. Each template uses `{{var}}`, `{{#each}}…{{/each}}`, and `{{#if}}…{{/if}}` — three forms, no nesting.
- **`src/components/`** holds the MDX component templates: `Dropcap.html`, `Pullquote.html`, `PullquoteNoCite.html`, `MarginaliaPin.html`, `Footnote.html`. Each is a tiny HTML fragment with `{{prop}}` substitutions, resolved when an MDX file references the component.
- **`src/css/`** is three files: `tokens.css` (the design system — palette, type, spacing, motion), `kit.css` (reusable UI components — buttons, cards, nav, footer, hero, reader shell), and `site.css` (everything else — page-specific blocks, marginalia, drawer, the dark-mode quirks).
- **`src/js/`** is the runtime — five tiny scripts that I'll describe in a minute.

The build is the act of folding `src/` into `dist/`.

## The build

`build.js` does, roughly, this, in order.

It reads `site.json`, `now.json`, `about.json` — the structured content. It reads every MDX file under `src/content/` into memory. For each MDX file it splits frontmatter from body, parses the frontmatter (a YAML-lite parser that handles strings and arrays-of-objects, which is all the frontmatter ever needs to be), and runs `resolveComponents` on the body. That step substitutes `<Dropcap text="…" />` and friends for their HTML templates from `src/components/`, before the markdown parser runs, so the parser sees real HTML and leaves it alone.

<Pullquote text="The markdown parser is forty lines. It handles paragraphs, headings, lists, blockquotes, horizontal rules, links, bold, italic, and inline code. That is the only markdown this site needs." cite="— from the source, lines 124–193 of build.js" />

It auto-slugs every `##` heading into an `id` so the table-of-contents anchors resolve. It then renders the layout for each page through a tiny template engine that walks the templates and replaces `{{var}}` with values from the data, runs `{{#each items}}…{{/each}}` blocks against arrays, and gates `{{#if name}}…{{/if}}` blocks on truthiness. The engine doesn't support nested loops; when a page needs them, the outer loop is pre-rendered to HTML in `build.js` and passed in as a single variable. This is a shape limitation that turns out to keep the templates honest — when a layout starts wanting nested loops, the right answer is usually a smaller component, not a smarter template.

It concatenates the three CSS files in order — `tokens.css` then `kit.css` then `site.css` — into a single `dist/css/style.css`. It concatenates the five JS files in order into `dist/js/main.js`. It writes one `index.html` per page into the right directory under `dist/`.

That's the build. On my laptop it runs in about 50 milliseconds for 80 pages. On the iPhone — iSH, no JIT — it's 200 milliseconds. Either way, by the time you've moved your hand, the site has rebuilt.

## The JavaScript

There's a runtime, but it does almost nothing. Five files, ~150 lines together.

- **`theme.js`** — flips a `data-theme="dark"` attribute on `<html>` based on the user's last choice in `localStorage`. The visual swap (sun icon vs. moon, palette inversion) is all in CSS, keyed off that attribute. It runs synchronously before paint to avoid the dark-mode flash, and the click handler is six lines long.
- **`nav.js`** — opens and closes the menu drawer. It's the longest of the five files at ~80 lines, and the length is mostly accessibility plumbing: it traps focus inside the dialog while it's open, marks the rest of the page `inert` and `aria-hidden` so screen readers don't wander into the background, returns focus to the trigger on close, and listens for `Escape`. None of that is interesting individually; together it's the minimum a modal dialog has to do to be honest.
- **`accessibility.js`** — three lines. It listens for the first `Tab` keypress and adds a `using-keyboard` class to the body. The CSS uses that to refine focus rings. After the first mouse click it removes the class.
- **`texture.js`** — two lines. It applies the warm paper noise texture to the body once on load. Could be CSS-only; lives in JS so the noise can be turned off for users who prefer reduced motion (it isn't yet, but the hook is there).
- **`floorboards.js`** — two lines. It exists because the site's tagline calls the JavaScript "creaky floorboards I coded" and I wanted at least one creaky floorboard. It logs a single line to the console.

Concatenated, the runtime is ~3 kilobytes. There is no framework. There is no router — every page is a real HTML file at a real URL. There is no hydration step. There is no client-side rendering.

When you click a link, the browser fetches the next HTML file. When you reload, you get the same HTML the build wrote. When you view source, you see the entire page. When you hit Pages on a Sunday morning and the server is slow, the browser shows you the cached HTML it already has, because there is nothing else for it to wait for.

## The payoff

The payoff is that the site builds anywhere Node runs. It builds on a desktop. It builds on a laptop. It builds on a phone, in a Linux emulator, on the train. It builds on GitHub Actions in 4 seconds, because the slowest part is `actions/checkout`. It builds when the wifi is bad, because there's nothing to download. It builds in 2030, because `fs`, `path`, and standards-track CSS will still be there.

There are real costs. I do not get hot module reload, fancy syntax highlighting, image optimization, or RSS without writing it myself. I do not get type checking on the templates, so a typo in `{{ownerName}}` becomes an empty string in the rendered HTML. I do not get any of the things a modern toolchain gives you, because I am not running a modern toolchain.

For a personal site, those trades have all gone the way I wanted. The thing I most enjoy about working on this codebase is that I never have to think about the build. There is nothing between me and the writing, and on the days when the writing is the hard part — which is most days — that turns out to matter.

If you have been meaning to start a personal site and you have been put off by the toolchain, consider this a small note in the margin: you can make one with `fs`, `path`, three CSS files, and the patience to write a forty-line markdown parser. It even builds on your phone.
